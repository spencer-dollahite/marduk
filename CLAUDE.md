# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What is Marduk

Marduk is an assistive technology platform for macOS (targeting macOS 26) that replaces VoiceOver. It combines native Apple Accessibility APIs, spatial audio, and Karabiner Elements with Vim-style modal navigation. It runs as a background daemon with no UI — all configuration is via files and CLI.

## Build & Run

```bash
swift build                          # Build (output: .build/debug/marduk)
marduk start                         # Start daemon (via launchd if installed, else foreground)
marduk start --foreground [--debug]  # Run inline in the terminal (needs Accessibility permission)
marduk stop                          # Stop daemon (stays stopped; agent restarts it at next login)
marduk install                       # Install launchd agent: login autostart + crash restart
marduk uninstall                     # Remove the launchd agent
marduk status                        # Daemon / agent / launchd / log status
marduk update                        # Git pull + build + hot-reload (run from project dir)
marduk speak "text"                  # Speak text (forwards to daemon if running, else inline)
marduk speak --debug "text"          # Speak with ducking debug logs
marduk config                        # Show config JSON
marduk config rate <50-360>          # Set speech rate in WPM (`:config pitch <50-200>` = reading-voice pitch %)
marduk voices --test                 # Interactive voice tester
```

No external dependencies — pure Swift Package Manager with native Apple frameworks only.

## Architecture

```
main.swift (CLI argument parser)
  ↓
DaemonServer (Unix socket IPC, per-user $DARWIN_USER_TEMP_DIR/marduk.sock)
  ├── SpeechEngine (AVSpeechSynthesizer — dual voices: reading + announcement; auto-select is IDENTITY-FIRST: the system's default voice for en-US (`AVSpeechSynthesisVoice(language:)` — reflects the user's Spoken Content choice; Marduk users are existing AT users whose familiar voice beats any quality ranking) upgraded within the SAME NAME to the best installed edition (compact Samantha → enhanced Samantha), falling back enhanced > premium > compact only when there's no system default. Premium voices trade articulation for naturalness and degrade at high rates (field-tested), so they're OFFERED (welcome line, :voices download hint, tip) never imposed; user-downloaded in Settings > Accessibility > Read and Speak Content, no API can fetch them; an explicit voiceIdentifier always wins)
  │   └── AudioDucker (probes playback state, ducks external audio during speech)
  ├── KeyboardMonitor (CGEventTap — Vim modal: NORMAL/INSERT)
  └── ModeOverlay (click-through NSWindows — opt-in mode border + pointer dot, OFF by default)
```

**Threading model:** Main thread runs RunLoop for AVSpeechSynthesizer callbacks + CGEventTap. Socket accept runs on background DispatchQueue. AudioDucker operations serialize on its own queue. AppleScript calls are synchronous via `Process`.

**IPC protocol:** Text-based commands over Unix socket (e.g., `speak Hello`, `rate 0.5`, `ping` → `OK pong`). `DaemonClient.send()` is synchronous request-response.

**Config:** `~/.config/marduk/config.json` — `MardukConfig` (Codable) with ducking, speech, display, keyboard, verbalizer, update, and overlay settings. `ConfigLoader` auto-creates defaults on first run. Speech/keyboard/verbalizer settings apply LIVE via `:config <key> <value>` (DaemonServer retains `config`, mutates, saves); hand-editing the file still requires a restart. **Adding fields:** new keys inside existing blocks (and new top-level blocks) must be declared Optional and defaulted at the consumption site with `??` — a required key missing from the user's existing config.json fails the whole decode, and `ConfigLoader.load()` responds to a failed decode by silently resetting the file to defaults, wiping the user's voice/rate settings.

## Key Patterns

- **Design doctrine — ladders + tables:** every capability is a LADDER of platform-generic strategies tried in order (R: AX value → container descent → PDF-by-file → web-area walk → scripted fallback); a quirk discovered via one app is implemented against the platform PATTERN it represents (Notes → wrapper-focus descent for all apps; Terminal → visible-range validation for all text apps; PT → holder-classing for all flappy apps; Qt → untitled-dialog filter). Where macOS forces per-app knowledge it lives in DATA — `pageChords`, `builtInInvertPrefixes`/`flappyPrefixes`, `mediaKeyClientBundles`/path tokens, `systemAgents`, `scriptedTextFallbacks` — never in code paths. **The test for new work: a new app should cost a table row; a new platform pattern costs a ladder rung; an app-specific code path needs a written justification** (current justified one-offs: Firefox `n`/`8` drives Firefox's OWN Narrate; Preview's dark-appearance press already generalizes by menu-title SEARCH).
- **Logging:** `fputs("[component] message\n", stderr)` — components: `[speech]`, `[ducker]`, `[keyboard]`, `[marduk]`, `[update]`, `[main]`, `[display]`, `[overlay]`, `[earcon]`, `[verbalizer]`, `[agent]`, `[sign]`, `[command]`, `[tutorial]`, `[bundle]`. **Privacy (allowlist, not redaction):** NEVER log user content — selected/spoken text, read-search queries — only lengths and metadata (char counts, keycodes, AX error codes, paths); the log is DESIGNED to be pasted into public GitHub issues (`:log copy`, `:bug`). Colon-command echoes (`: config rate 100`) are structured vocabulary, fine. Sole content exception: the inline `marduk speak --debug` path echoes its own CLI argument to the invoking terminal (never the daemon log). Don't add SSN/address regex scrubbing — keep content out at the source instead.
- **Sendable workarounds:** `@unchecked Sendable` classes with `nonisolated(unsafe)` for AV framework types
- **Callbacks:** `onEvent` pattern (onSpeak, onStop, onAnnounce, onWordBoundary); speech chaining uses the per-utterance `completion:` parameter of `speak`/`announce`
- **Enums as namespaces:** `Earcon`, `DaemonClient`, `ConfigLoader`, `MardukDaemon` are caseless enums with static members
- **AppleScript execution:** Via `Process` + `/usr/bin/osascript`, three variants: `runAppleScript` (→ Int?), `runAppleScriptString` (→ String?), `runAppleScriptSync` (fire-and-forget)
- **Key suppression:** In NORMAL mode, alpha keys are withheld by the typing-rescue burst buffer (~300ms) and then either executed as commands, beeped, or replayed into the app as typing — except `k`, which passes straight through, and `i`, which enters INSERT instantly. Space, numbers, arrows, function keys pass through (but see Space pause/resume below). Cmd/Ctrl combos always pass through. With `keyboard.typingRescue: false`, alpha keys are consumed immediately (pre-rescue behavior).

## Speech Preprocessing (Sanitizer + Verbalizer)

**System pronunciations (macOS dictionary import):** macOS's own pronunciation editor (Settings → Accessibility → Read and Speak Content → Pronunciations) never applies to third-party AVSpeechSynthesizer clients (hardware-verified on macOS 26), so `SystemPronunciations` (Sources/Audio/SystemPronunciations.swift) imports the store directly: defaults domain `com.apple.universalaccess`, `pronunciationsEnabledKey` + `pronunciationsListKey` — an array of NSKeyedArchiver blobs of the private `UAPronunciationSubstitutionModel` (fields: originalString, replacementString, ipa attributed string — empty for typed entries, the phonetic transcription for voice-captured ones — active, ignoreCase, language, appliesToAllApps, bundleIdentifiers). Decoded via a stand-in class mapped with `setClass(forClassName:)`, secure coding off, `.setErrorAndReturn` — UNDOCUMENTED FORMAT, so every failure degrades to zero entries (count-only logging; phrases are user content, never logged). Fetched fresh via CFPreferences at every read start (Settings edits apply to the next read; `speech.frontmostAppProvider` supplies the cached frontmost bundle ID for per-app scoping, language prefix-matches the reading voice). TEXT entries substitute whole-word into the raw read before preprocessing (so motions/search/spell see one consistent text — `z` spells the replacement, accepted); IPA entries never mutate text — `makeReadUtterance` applies `AVSpeechSynthesisIPANotationAttribute` over whole-word matches (offsets untouched; respeak re-matches its substring). `announce()`/echo/hover are exempt. `:pronunciation` deep-links to the pane. Tests: SystemPronunciationsTests (fixtures are byte-faithful plistlib reconstructions of the real archive).

`SpeechPreprocessor` (Sources/Audio/SpeechPreprocessor.swift) runs inside `SpeechEngine.speak()` on every read: **INPUT CAP first** (60k UTF-16 units, cheap gate before any pass — field incident: R in Terminal delivered a 9.1M-char scrollback and the O(n) passes froze the main thread for seconds, starving the EVENT TAP and freezing keyboard input system-wide; output is capped at 50k anyway, so bigger input is pure wasted time; count-only log) → **sanitize** (strips control chars, invisible Unicode — ZWSP/BOM/variation selectors/directional marks — private-use chars; emoji ZWJ sequences survive; NBSP → space) → **hash abbreviation** (word-bounded all-hex runs at standard digest lengths — 32/40/64/128 → md5/sha1/sha256/sha512, named by length convention — collapse to "md5 ending in 2 7 e"; requires ≥1 digit and ≥1 letter so numeric IDs don't mislabel; runs at every level, `verbalizer.hashes: false` disables) → **identifier splitting** (`splitIdentifiers` — camelCase humps and internal snake_case underscores become spaces: "parseHTMLBody_v2" → "parse HTML Body v 2", acronym runs split before a following word — XMLHttp → "XML Http"; a token qualifies only with an internal underscore or a case hump alongside lowercase, so plain words, ALLCAPS acronyms, UTF16-style tokens, and `__init__` dunders pass untouched; own toggle like hashes — every level, `verbalizer.identifiers: false` / `:config identifiers off` disables) → **verbalize** (speaks symbols as words per `verbalizer.level`: `none`/`some`/`most`/`all`, default `most` = code symbols spoken, prose punctuation left to natural pausing; digraphs like `->` "arrow" and `!=` "not equals"; `#` is context-spoken: all-digit tail → "number" (item #66), any attached word/mixed/symbol tail → "hashtag" (#topic, #tag2you, #!), standalone → "hash" — a user override of `#` wins in every context; symbol runs ≥3 collapse to "N name" using the full symbol table even for symbols unspoken at the level — `-----` says "5 dash" at most; a bare `...` stays raw below `all` (natural ellipsis pause); unnamed symbols cap at 3 raw) → **whitespace normalize**. Sanitize is the anti-bail stage and runs at every level. Empty-after-processing skips speaking but still fires the `completion:` (the inline CLI blocks on it). `announce()` gets sanitize only. Config: `verbalizer` block — `level`, plus `symbols` overrides (`{"*": "asterisk", "->": "maps to", "%": ""}`; empty string silences a symbol). Config is read at daemon init; `:config level|hashes|identifiers` rebuilds the preprocessor live, while hand-edits to the file still require a restart. Tests: `swift test` (Tests/MardukTests, macOS only).

## Audio Ducking Flow

`prepareToDuck()` (before speech) → probes the CoreAudio-based mediaKey state synchronously (so our own speech can't contaminate it), stores it on the ducker queue → `duck()` (async, on didStart) → uses cached state, probes Apple Music/Spotify via AppleScript on the ducker queue (never on the main thread), only pauses what's actually playing → `unduck()` (async, on didFinish/didCancel) → only restores what was ducked. `audioProducingPIDs()` filters out our own PID AND Apple's speech-synthesis service process (enhanced voices render there — without the filter, back-to-back reads mistake draining speech audio for playing media). **The media key is only sent to a claimed room:** `.mediaKey` counts as playing ONLY when an audio-producing process is a media-key CLIENT (`audibleMediaKeyClientExists` — bundle-ID list ∪ executable-path tokens, because browser audio renders in helpers: Chrome → "Google Chrome Helper", Safari → com.apple.WebKit.WebContent; extras via `ducking.mediaKeyApps`). An UNCLAIMED play/pause press makes macOS launch Music (first-external-user report) — calls/games/unknown audio are therefore spoken over, never paused. All osascript runs have a kill-on-timeout watchdog so a hung System Events can't wedge the ducker queue.

**Important:** Never use fixed delays to wait for speech to finish. Pass a `completion:` to `speech.speak`/`speech.announce` to chain actions after speech completes — this ensures unduck() has time to restore media before the next action. Completions are tracked per-utterance (fired on finish or cancel, only for their own utterance), so a replaced utterance can never fire or clear its successor's callback. Only the current utterance's end unducks; stale didCancel deliveries are ignored.

## Keyboard Modal System

- **NORMAL** (default): Commands active, letter keys suppressed. `s` toggles MARDUK-NATIVE pointer hover speech (`HoverSpeech`, Sources/Input/HoverSpeech.swift): pointer POLLING (~8Hz Timer, only while active; NEVER NSEvent global monitors — field 2026-07-22: one installed cleanly and delivered ZERO events to the background daemon, hover spoke once then went deaf; a tripwire test fails if they return) → settle ~0.3s via the pure `HoverDwell` (clock-injected, tested — sub-threshold drift accumulates against the rest point so slow glides keep naming controls) → AX element-at-position → speak the element's NAME (title ?? description ?? value, minimal verbosity by founding rule) in the READING voice/rate/pitch over the echo channel — no Settings shortcut setup, no media pausing (BY DESIGN — short labels must not thrash playback), never disturbs a read; per-stage session tally logged on toggle-off (counts/lengths only); same-element/same-label repeats and Marduk's own windows stay silent; deactivated on Ctrl+Option+M off and teardown. (Replaced the old macOS speak-items-under-pointer shortcut integration, whose separately-configured system voice never matched.)
- **Typing rescue** (NORMAL): unmodified letter keyDowns are withheld ~300ms (`keyboard.typingBurstMs`; `keyboard.typingRescue: false` disables) instead of executing immediately. A quick burst containing a non-command letter means the user forgot the mode and is typing — auto-switch to INSERT, play a falling sweep (`Earcon.fallToInsert`), and replay the withheld keys synthetically as down+up pairs (marker-tagged, in order — pairs because the real keyUps usually passed through before the replay posts; a replay-rollover guard swallows and re-queues keys typed during the switch so nothing reorders). All-command bursts (`s` then `r`) execute as commands on timer expiry via recursive redispatch through `handleEvent` (mode-aware: a buffered `i` flips to INSERT and later buffered keys get typed, not executed). Consequences: single letter commands fire ~300ms late; `i` on an empty buffer is instant (the i-then-type flow); `v`+motion (`vj`, `v3j`) flushes immediately and stays instant; `tt` resolves on the second t (commands buffered before the pair flush first); words starting with `i` lose their leading i; a command followed quickly by `k` reads as typing (words like "skip" must rescue); with rescue disabled there is no `tt` (each `t` speaks time immediately).
- **INSERT** (`i`): Entry always plays `Earcon.fallToInsert` (explicit `i` and typing rescue sound identical). All keys pass through. Escape is tap/hold: a tapped Escape is delivered to the app on key release (synthetic re-post — vim/Claude Code keep their Escape); holding it past the threshold (default 400ms, `keyboard.escapeHoldMs` in config) returns to NORMAL and plays a rising sweep (`Earcon.riseToNormal`) — the app never sees a held Escape. The withheld press is flushed early if another key rolls over it, so fast Esc+j typing stays ordered. Modified Escapes (Cmd/Ctrl/Shift) pass through untouched.
- **VISUAL** (`v`): Character-wise selection with hjkl, `r` reads selection. Entry is spoken ("visual"/"visual line"); exit via Escape/`v` plays `Earcon.riseToNormal` (`r` exits silently — the read is the feedback)
- **VISUAL LINE** (`V`): Line-wise selection, auto-selects current line on entry
- **COMMAND** (`:` = Shift+; in NORMAL): vim-style command line, driven entirely by the event tap. Chars append to `commandBuffer` (keycode→char via `commandKeyChars`; echo per char when `commandEcho`), Return submits via `onCommandSubmit` (empty = cancel), Escape cancels, Delete edits (exits on empty buffer), Tab autocompletes to the palette selection via `onCommandTab`→`replaceCommandBuffer`, Up/Down move the selection (Ctrl+N/Ctrl+P are vim-native synonyms — the second Ctrl carve-out, COMMAND-only; passing them through would emacs-move the app's cursor mid-command), `?` speaks the current options, a ~1.5s typing pause speaks them automatically (`onCommandIdle`; silent when none), and `/` on an empty buffer starts an fzf-style FUZZY SEARCH over the whole catalog (commands + config keys; greedy-subsequence `fuzzyScore`, lower = tighter; Return accepts the selection — commands execute, keys expand into the value stage; `/` buffers never auto-resolve). Cmd/Ctrl AND Option combos pass through (the user's zoom shortcuts ride on Option — command mode must never eat them), as do unmapped non-typing keys (F-keys, keypad, custom codes); only unmapped typing-shaped keys (`typingPunctuationKeys`) buzz, logging the keycode. Autorepeat suppressed except Delete/arrows. Commands (unique-prefix expanded, vim-style — `:conf ra 230` works): `:help`/`:h`, `:commands`/`:c`, `:tutorial` (toggles the interactive tour — Tutorial.swift watches real mode/read/pause events), `:tip` (random feature tip from HelpText.tips, no immediate repeats), `:config`/`:set <key> <value>` (live apply + persist), `:pronunciation` (opens the system pronunciation editor pane — see System pronunciations), `:typing` (opens the same pane for macOS typing feedback — front-desk routing to features Apple already built; announcement also names Marduk's own `echo`), `:voices` (voice picker: expands to a `voices ` stage listing installed English voices best-quality-first; typed text fuzzy-filters; arrow selection previews each candidate IN that voice via `announce(_:voice:)`; Return/`/`-accept applies to `speech.voice` + persists `voiceIdentifier`; the tap's Return handler keeps COMMAND mode alive for `voices`-prefixed buffers exactly like `/`, and the daemon intercepts them before parse — `.voices` in the parse switch is compiler-exhaustiveness only). **Auto-accept (dmenu semantics):** ~350ms after the last typed char (`ColonCommand.autoResolve`, debounced in `handleCommandChange` so fast full-word typing is never cut off), an unambiguous buffer acts without Enter — argless commands and unique enum values execute, `config`/keys expand and advance; numbers always need Enter; deletions never auto-accept (canAutoAccept flag on `onCommandChange`), or removing an auto-added space would re-add it. Settings table lives in `ColonCommand.settings` (shared by parser, palette, and validation). The **command palette** (Sources/Display/CommandPalette.swift, gated by `keyboard.commandPalette`) is a dmenu-style panel whose INPUT still comes from the event tap — it renders the buffer + candidates from `CommandCompleter` — but it TAKES key focus while open (Spotlight-style: remembers `NSWorkspace.frontmostApplication`, `NSApp.activate()`, `makeKeyAndOrderFront`; borderless panels need the `canBecomeKey` override) and reactivates the previous app on hide. Position is configurable (`keyboard.palettePosition` / `:config position pointer|center`): `pointer` (DEFAULT) opens at the cursor — the ONLY zoom-proof placement, since zoom always keeps the real cursor in view; `center` is fully centered on the pointer's screen. Zoom focus-following can NOT be made to track the palette — exhaustively disproven (bare binary, signed bundle, .regular policy, real focused caret, every AX notification, every form of synthetic pointer input; no viewport API exists — the old border overlay's lesson). Do not reopen that investigation. The prompt line is a REAL focused `NSTextView` (PromptTextView — keyDown/insertText swallowed, the tap owns input) with a genuine blinking caret, so zoom's follow-focus/text-insertion modes track the palette natively via AX notifications; level is `.floating`, NOT `.screenSaver` (the aggressive level broke zoom state while open); rows are mouse-clickable (click = Tab on that row, `onRowClick`). With the palette off, command mode is pure audio and steals nothing. `DaemonServer.run()` initializes `NSApplication.shared` (`.accessory`; `:config dock on` / `display.dockIcon` flips to `.regular` LIVE — Dock + app switcher + Force Quit window are ONE policy on macOS, no Force-Quit-only state exists) for it.

AX failures log their error code; -1001 = the timeout
- **Firefox narration handoff (`n`, `8`):** in NORMAL with Firefox frontmost, `n` stops Marduk's speech, pauses media via the ducker, sets a **duck hold** (`AudioDucker.holdDucking` — unduck() becomes a no-op so Marduk's own announcements can't resume music mid-narration), then posts a synthetic `n` to Firefox (~0.35s later) which drives Reader-mode Narrate's play/pause (`startNarrationHandoff`). Second `n`, Escape (NORMAL), or Ctrl+Option+M posts `n` again and `releaseHoldAndUnduck()`. STARTING requires `narrationContext()` — an AX walk UP from the focused element to the nearest AXWebArea checking for an `about:reader` URL (main queue only, 0.5s timeouts); this is what makes INSERT-mode `n` safe: it always passes through (it IS Narrate's toggle) and only duck/holds when focus is inside a Reader document, so text boxes and the URL bar keep their n's. `8` (NORMAL, Firefox, keycode 28 — the one exception to digits-pass-through) is the combo key: posts Cmd+Option+R, polls `narrationContext()` (~2.5s), then hands off; `8` while narrating = stop + release + close Reader. Known limit: state can desync if the user drives Narrate from Firefox's UI mid-handoff (n is a blind toggle).
- **Speed keys (opt-in, OFF by default):** `:config speedkeys on` / `keyboard.speedKeys` — Option+Up/Down nudge the speech rate ±10 WPM in NORMAL/VISUAL only (INSERT/COMMAND excluded: apps own Option+arrows there; Shift combos pass through). Deliberately NO autorepeat guard — holding glides. Rate applies per event; announcement + config write debounce 500ms after the last nudge (`rateSaveTimer`). Setting key is `speedkeys`, NOT `ratekeys` — `rate` would be its prefix, breaking unique-prefix expansion (guarded by `testNoSettingKeyIsPrefixOfAnother`).
- **Visual mode overlay (opt-in, OFF by default):** `ModeOverlay` (Sources/Display/ModeOverlay.swift) — screen-edge border and/or pointer-following dot colored by mode: red NORMAL, green INSERT, blue VISUAL, purple READING (`overlay.readingColor`, default #AF52DE — READING capture OVERRIDES the mode color via `setReading`, driven by `KeyboardMonitor.onReadingChange` from `readingCapture`'s didSet, and `rebuildOverlay()` re-applies it so a live `:config thickness` mid-read stays purple); COMMAND keeps normal's color (the palette is its own indicator); `"none"` as a color hides that mode. `:config border on|off`, `:config pointer on|off`, `:config thickness <1-40>` apply live (`rebuildOverlay()` — full teardown + recreate); colors and `pointerSize` are file-only (`overlay` config block, all keys Optional). Marduk-off hides everything (`setEnabled` via `onEnabledChange`). Zoom caveat: fullscreen zoom magnifies the composited framebuffer, so the border can be entirely out of the zoomed view — the pointer dot is the zoom-proof indicator (zoom always tracks the pointer). Windows are `.screenSaver` level, click-through, never take key focus (safe, unlike the palette).
- **Space (read pause/resume):** while a content read is speaking or paused (`SpeechEngine.readActive` — announcements excluded), unmodified Space in NORMAL/VISUAL pauses IMMEDIATELY / resumes (never pauseSpeaking(at:.word) — a deferred pause landing after a stop wedges the synthesizer mute and kills didStart-driven ducking; field-diagnosed), and never reaches the app. In INSERT, Space is always a real space, even mid-read. Otherwise Space types/passes as normal. A typing-rescue burst in flight wins over pause (user is typing). Escape in NORMAL cancels a paused read (a paused synthesizer still reports `isSpeaking`), freeing Space back to normal. Media stays ducked/paused across a speech pause — only the read's end or Escape (stop) unducks. `readActive` is plain stored state so the tap callback can read it synchronously (no AV query in the callback).
- **Read motions / READING capture (ON by default; `:config readmotions off` / `keyboard.readMotions` disables):** while enabled, an active read (`readActive`) CAPTURES the keyboard from ANY mode (`readingCapture`, set synchronously by `SpeechEngine.readActive`'s didSet → `onReadActiveChange` → `readStateChanged`; entry skipped in COMMAND — post-`:` confirmation reads must not steal the palette). READING is a real mode: `i` and held Escape are the ONLY exits — `i` → INSERT with the falling sweep while the READ KEEPS PLAYING (user-redesigned: type notes while listening; the capture suspends, no readActive edge); held Escape from that INSERT RECLAIMS the capture (climbs INSERT → READING → NORMAL one level per hold — the rung is audible: reclaiming READING plays `Earcon.riseToReading` (ends a whole tone LOWER, 784), reaching NORMAL plays `Earcon.riseToNormal` (ends higher, 990); Option+Escape stops audio from anywhere; a NEW read via onNewRead also reclaims); Escape is tap/hold (`pendingReadingEscape`, same `escapeHoldThreshold` + `escapeHoldFired` keyUp-swallow as INSERT's): a TAP toggles pause/resume exactly like Space (resolved on keyUp; a rollover key resolves it as a tap first), a HOLD stops the read → NORMAL with the rising sweep; every other typing-shaped key (letters incl. `k`, stray digits, punctuation, Return/Tab/Delete) buzzes instead of leaking into the app — a read fired from INSERT used to let `(`/`gg` type over the selection. ## Release Pipeline

`scripts/release.sh <version>` (run on the Mac, repo root — or spoken end-to-end via `dd`, which auto-bumps the PATCH and spawns this same script; minor/major bumps stay manual) produces a notarized downloadable release: bumps Version.swift → commit/tag/push → **waits for CI green** (`gh run watch`) → `swift build -c release` → `marduk bundle` (internal CLI command: assembles Marduk.app from the invoking binary, prints the exec path) → **distribution re-sign** (the Bundler's own signature is development-grade `--timestamp=none`; notarization requires `--options runtime --timestamp` + the `com.apple.security.automation.apple-events` entitlement so media control survives the hardened runtime) → notarize the APP (zip submit, `marduk-notary` profile) → staple the app → build a drag-to-Applications DMG (`hdiutil`, /Applications symlink) → sign + notarize + staple the DMG too (belt-and-braces: both artifacts verify offline) → `gh release create` with the DMG and commit-subject notes → **update the Homebrew tap** (`spencer-dollahite/homebrew-marduk`): rewrite `Casks/marduk.rb` with the new version + local DMG sha256 and push (runs before the local DMG is deleted; the cask pins the VERSIONED asset URL `releases/download/v<version>/Marduk.dmg`, never the floating `latest` link — hash pinning breaks otherwise). Users: `brew install --cask spencer-dollahite/marduk/marduk`. **Release installs** (bundle outside any repo): `marduk install` detects `/Marduk.app/Contents/MacOS/` with no project dir and installs AS-IS (reassembling/re-signing would destroy the notarized signature); the update paths detect no-project-dir and check the latest GitHub release tag instead (`checkLatestRelease` + pure `ReleaseCheck.parseLatestTag`, unit-tested): manual `u` speaks "Marduk X is available" / "up to date", the periodic timer announces each new version once (`lastAnnouncedRelease`); the install hint is channel-aware — `InstallChannel` detects Homebrew via the Caskroom dir and says "run brew upgrade", plain releases get "download from GitHub" (release-channel self-INSTALL = future work). `:bug` prefills the issue form's setup field with version + channel + macOS version. **Finder onboarding:** double-clicking Marduk.app (no CLI args, bundle path, no repo) runs `install`; if already running it speaks "already installed"; after install with no Accessibility grant it opens the Privacy_Accessibility Settings pane while the daemon's spoken self-heal explains the grant. Download→drag→double-click→follow-the-voice is the entire release onboarding.

## Progressive Onboarding

`Onboarding` (Sources/App/Onboarding.swift) + `OnceMarker` (Sources/App/OnceMarker.swift) run the respectful, spread-out teaching layer beside the opt-in `:tutorial`. **`OnceMarker`** consolidates the once-ever dotfile idiom (`~/.config/marduk/.NAME` → fileExists → write) that was copy-pasted at six sites — `seen`/`mark`/`clear`/`firstTime`; all markers (welcomed, pdfdark-noticed, ke-noticed, macos-N-noticed, dialogfocus-explained, zoomfollow-hinted, tutored) route through it. **`Onboarding`** is the pacing brain: `offer`/`offerQuestion` (feature hints + first-use config questions) fire only through the pure, tested `shouldSurface` gate — once-ever, a per-session cap (`sessionCap`), a `cooldown` between interjections, quieting once `experienced` (lifetime `hintsShown ≥ experiencedThreshold`, persisted to `config.onboarding`, OR the `.tutored` marker written by `tutorial.onComplete`), and NEVER while a read is speaking (a hint that interrupts a read is worse than one never shown). `notice` handles critical items (Karabiner absence, untested macOS) — bypasses pacing but still once-ever + yields-until-quiet (generalizes the old retry loops). `:config hints on|off` is the master switch. **First-run gateway:** the welcome (`HelpText.welcome`, spoken once via the read path) ends ON a one-key choice armed in its completion — `t` = start `:tutorial` now, `p` = progressive hints on, `s` = advanced/silent (`handleWelcomeChoice`); Escape/timeout/any other key leaves progressive hints on (the respectful default). Uses the generalized `armQuestion(keys:)` (t/p/s). Phase 2 (event-driven feature hints via the Tutorial event stream) and Phase 3 (the broader y/n config-question set) build on this foundation.

## Safe Mode (BootGuard)

`BootGuard` (Sources/App/BootGuard.swift): rapid-crash detection — a daemon that crashes at STARTUP can never self-update its own fix (the one hole in the update safety net, flagged in the macOS-27 readiness audit). `.boot-attempts` counts boots within a 5-min window (KeepAlive loops are ~10s apart); at 3, `Daemon.run` boots SAFE MODE: speech, socket, event tap, Karabiner handoff, and the UPDATE TRAIN stay up; DialogSentinel, DisplayInverter, and ModeOverlay are skipped. Spoken announcement at boot (+8s); `ping` answers `OK pong safe` and `marduk status` surfaces it. Surviving 60s — or any clean teardown — marks stable. Companion: `announceUntestedMacOSOnce` speaks once per macOS major above `Marduk.testedMacOSMajor` (bump after each major's checklist passes; see memory macos-27-readiness).

## Running as a Service (launchd)

`Bundler` (Sources/App/Bundler.swift, `[bundle]` tag) assembles `<repo>/Marduk.app` on the Mac at install/update time — Info.plist via PropertyListSerialization, the M icon DRAWN IN CODE (CoreGraphics, same path as assets/logo.svg, iconutil-packed; failure = iconless bundle, non-fatal), binary copied in, bundle signed, then ATOMIC SWAP (`Marduk.app.new` → aside → in; never modify the live bundle in place — a running daemon keeps its inode). **Identity trio, keep identical:** launchd label == `Codesign.identifier` == CFBundleIdentifier == `com.marduk.daemon` (identical designated requirement is what can carry TCC grants across the bare-binary→bundle migration). `LSUIElement` keeps it out of the Dock. Bare-binary operation (`start --foreground` debugging, binary copied out of the repo) still works everywhere — bundle-ness is detected, never assumed. SMAppService adoption = future work.

**Migration (bare binary → bundle):** detected by `LaunchAgent.installedProgramPath() != bundleExec`. The CLI update rebootstraps synchronously (safe — it isn't the daemon). The daemon-side `uu` CANNOT bootout itself (deadlock: launchd waits for our exit while we wait for launchctl) — it writes the new plist FILE only, spawns `LaunchAgent.relaunchDetached()` (a `/bin/sh` helper via `posix_spawn` + `POSIX_SPAWN_SETSID` — mandatory, because the OLD plist lacks `AbandonProcessGroup` and launchd SIGKILLs the daemon's process group on exit), announces the one-time migration (even on the silent auto-update path), and exits CLEAN (0) so KeepAlive can't race-relaunch the old path before the helper's bootout→bootstrap→kickstart lands. New plists carry `AbandonProcessGroup: true`.

`LaunchAgent` (Sources/App/LaunchAgent.swift) manages a per-user LaunchAgent: label `com.marduk.daemon`, plist at `~/Library/LaunchAgents/com.marduk.daemon.plist`, stderr/stdout to `~/Library/Logs/marduk.log` (truncated on install/start if >10 MB, and unconditionally on every SUCCESSFUL update just before restart — fresh log per build, which also ages out anything older logging policy wrote; the fresh log's FIRST line is a breadcrumb naming why (`log reset — restarting after …`, silent auto-updates labeled — truncation otherwise shreds the evidence of its own restart, field-diagnosed as an 'unexplained outage'; update FAILURES never truncate, preserving their diagnostics; no rotation). The plist runs `Marduk.app/Contents/MacOS/marduk start --foreground` so launchd supervises the daemon process directly. `KeepAlive = {SuccessfulExit: false}`: crash or non-zero exit → relaunch (launchd throttles rapid loops ~10s); clean exit 0 (`marduk stop`) stays stopped until next login or `marduk start`. When the agent is installed: `start` kickstarts the service (re-bootstrapping if needed), `update` (CLI) kickstarts instead of spawning, and the daemon-side `u` hot-update exits with code 75 after clean teardown so launchd relaunches the freshly built binary — a `Process` spawn would be unsupervised. `marduk start` exits 0 when the daemon is already running: under KeepAlive a non-zero exit from the agent-spawned instance would relaunch-loop. **TCC / signing:** install and both update paths sign the assembled BUNDLE (`Codesign.sign(bundleAt:)` — one call on the staging .app seals the nested executable; only ever called on `Marduk.app.new`, so no in-place hazard; `sign(binaryAt:)` remains for the bare-binary fallback and keeps its copy-sign-swap). Identity is certificate + `com.marduk.daemon`, now also the CFBundleIdentifier — the grant is bundle-anchored and survives rebuilds. If no keychain identity exists, signing is skipped with a loud log (plain `swift build` also skips assembly/signing — deploy via `marduk update`). If the tap dies after an update anyway, the daemon announces it and re-adding the Accessibility entry (pick `Marduk.app` in the file dialog) heals it live.

## Runtime Files

- `$(getconf DARWIN_USER_TEMP_DIR)marduk.sock` — daemon IPC socket (per-user /var/folders/…/T/; `MardukDaemon.runtimeDir` via confstr, NOT /tmp — world-writable /tmp let any local user drive the socket or squat the paths and block startup). Socket is chmod 0600 and `handleClient` getpeereid-rejects other UIDs. CLI and daemon share the path constants, so an updated CLI can't reach a pre-update daemon until its restart — one-time blip, `marduk start` heals.
- `$(getconf DARWIN_USER_TEMP_DIR)marduk.pid` — daemon PID file
- `~/.config/marduk/config.json` — user configuration (an unreadable file is preserved as `config.json.bad` before the defaults reset — never silently destroy settings; `.welcomed` / `.ke-noticed` / `.pdfdark-noticed` / `.boot-attempts` / `.macos-<major>-noticed` markers live beside it — first-run welcome, the once-only no-Karabiner onboarding hint, and the one-time dark-PDF explanation (`onDarkApplied`); `marduk status` also reports Karabiner presence)
- `~/Library/LaunchAgents/com.marduk.daemon.plist` — launch agent (when installed)
- `~/Library/Logs/marduk.log` — daemon log (when running under launchd)
- `<repo>/Marduk.app` — assembled bundle (gitignored; rebuilt by install/update)
