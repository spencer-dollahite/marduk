# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What is Marduk

Marduk is an assistive technology platform for macOS (targeting macOS 26) that replaces VoiceOver. It combines native Apple Accessibility APIs, spatial audio, and Karabiner Elements with Vim-style modal navigation. It runs as a background daemon with no UI — all configuration is via files and CLI.

## Build & Run

```bash
swift build                          # Build (output: .build/debug/marduk)
marduk start                         # Start daemon (via launchd if installed, else foreground)
marduk start --foreground [--debug]  # Run inline in the terminal (needs Accessibility permission)
marduk stop                          # Stop daemon (stays stopped; agent restarts it at next login)
marduk install                       # Install launchd agent: login autostart + crash restart
marduk uninstall                     # Remove the launchd agent
marduk status                        # Daemon / agent / launchd / log status
marduk update                        # Git pull + build + hot-reload (run from project dir)
marduk speak "text"                  # Speak text (forwards to daemon if running, else inline)
marduk speak --debug "text"          # Speak with ducking debug logs
marduk config                        # Show config JSON
marduk config rate <50-360>          # Set speech rate in WPM (`:config pitch <50-200>` = reading-voice pitch %)
marduk voices --test                 # Interactive voice tester
```

No external dependencies — pure Swift Package Manager with native Apple frameworks only.

## Architecture

```
main.swift (CLI argument parser)
  ↓
DaemonServer (Unix socket IPC, per-user $DARWIN_USER_TEMP_DIR/marduk.sock)
  ├── SpeechEngine (AVSpeechSynthesizer — dual voices: reading + announcement; auto-select is IDENTITY-FIRST: the system's default voice for en-US (`AVSpeechSynthesisVoice(language:)` — reflects the user's Spoken Content choice; Marduk users are existing AT users whose familiar voice beats any quality ranking) upgraded within the SAME NAME to the best installed edition (compact Samantha → enhanced Samantha), falling back enhanced > premium > compact only when there's no system default. Premium voices trade articulation for naturalness and degrade at high rates (field-tested), so they're OFFERED (welcome line, :voices download hint, tip) never imposed; user-downloaded in Settings > Accessibility > Read and Speak Content, no API can fetch them; an explicit voiceIdentifier always wins)
  │   └── AudioDucker (probes playback state, ducks external audio during speech)
  ├── KeyboardMonitor (CGEventTap — Vim modal: NORMAL/INSERT)
  └── ModeOverlay (click-through NSWindows — opt-in mode border + pointer dot, OFF by default)
```

**Threading model:** Main thread runs RunLoop for AVSpeechSynthesizer callbacks + CGEventTap. Socket accept runs on background DispatchQueue. AudioDucker operations serialize on its own queue. AppleScript calls are synchronous via `Process`.

**IPC protocol:** Text-based commands over Unix socket (e.g., `speak Hello`, `rate 0.5`, `ping` → `OK pong`). `DaemonClient.send()` is synchronous request-response.

**Config:** `~/.config/marduk/config.json` — `MardukConfig` (Codable) with ducking, speech, display, keyboard, verbalizer, update, and overlay settings. `ConfigLoader` auto-creates defaults on first run. Speech/keyboard/verbalizer settings apply LIVE via `:config <key> <value>` (DaemonServer retains `config`, mutates, saves); hand-editing the file still requires a restart. **Adding fields:** new keys inside existing blocks (and new top-level blocks) must be declared Optional and defaulted at the consumption site with `??` — a required key missing from the user's existing config.json fails the whole decode, and `ConfigLoader.load()` responds to a failed decode by silently resetting the file to defaults, wiping the user's voice/rate settings.

## Key Patterns

- **Logging:** `fputs("[component] message\n", stderr)` — components: `[speech]`, `[ducker]`, `[keyboard]`, `[marduk]`, `[update]`, `[main]`, `[display]`, `[overlay]`, `[earcon]`, `[verbalizer]`, `[agent]`, `[sign]`, `[command]`, `[tutorial]`, `[bundle]`. **Privacy (allowlist, not redaction):** NEVER log user content — selected/spoken text, read-search queries — only lengths and metadata (char counts, keycodes, AX error codes, paths); the log is DESIGNED to be pasted into public GitHub issues (`:log copy`, `:bug`). Colon-command echoes (`: config rate 100`) are structured vocabulary, fine. Sole content exception: the inline `marduk speak --debug` path echoes its own CLI argument to the invoking terminal (never the daemon log). Don't add SSN/address regex scrubbing — keep content out at the source instead.
- **Sendable workarounds:** `@unchecked Sendable` classes with `nonisolated(unsafe)` for AV framework types
- **Callbacks:** `onEvent` pattern (onSpeak, onStop, onAnnounce, onWordBoundary); speech chaining uses the per-utterance `completion:` parameter of `speak`/`announce`
- **Enums as namespaces:** `Earcon`, `DaemonClient`, `ConfigLoader`, `MardukDaemon` are caseless enums with static members
- **AppleScript execution:** Via `Process` + `/usr/bin/osascript`, three variants: `runAppleScript` (→ Int?), `runAppleScriptString` (→ String?), `runAppleScriptSync` (fire-and-forget)
- **Key suppression:** In NORMAL mode, alpha keys are withheld by the typing-rescue burst buffer (~300ms) and then either executed as commands, beeped, or replayed into the app as typing — except `k`, which passes straight through, and `i`, which enters INSERT instantly. Space, numbers, arrows, function keys pass through (but see Space pause/resume below). Cmd/Ctrl combos always pass through. With `keyboard.typingRescue: false`, alpha keys are consumed immediately (pre-rescue behavior).

## Speech Preprocessing (Sanitizer + Verbalizer)

**System pronunciations (macOS dictionary import):** macOS's own pronunciation editor (Settings → Accessibility → Read and Speak Content → Pronunciations) never applies to third-party AVSpeechSynthesizer clients (hardware-verified on macOS 26), so `SystemPronunciations` (Sources/Audio/SystemPronunciations.swift) imports the store directly: defaults domain `com.apple.universalaccess`, `pronunciationsEnabledKey` + `pronunciationsListKey` — an array of NSKeyedArchiver blobs of the private `UAPronunciationSubstitutionModel` (fields: originalString, replacementString, ipa attributed string — empty for typed entries, the phonetic transcription for voice-captured ones — active, ignoreCase, language, appliesToAllApps, bundleIdentifiers). Decoded via a stand-in class mapped with `setClass(forClassName:)`, secure coding off, `.setErrorAndReturn` — UNDOCUMENTED FORMAT, so every failure degrades to zero entries (count-only logging; phrases are user content, never logged). Fetched fresh via CFPreferences at every read start (Settings edits apply to the next read; `speech.frontmostAppProvider` supplies the cached frontmost bundle ID for per-app scoping, language prefix-matches the reading voice). TEXT entries substitute whole-word into the raw read before preprocessing (so motions/search/spell see one consistent text — `z` spells the replacement, accepted); IPA entries never mutate text — `makeReadUtterance` applies `AVSpeechSynthesisIPANotationAttribute` over whole-word matches (offsets untouched; respeak re-matches its substring). `announce()`/echo/hover are exempt. `:pronunciation` deep-links to the pane. Tests: SystemPronunciationsTests (fixtures are byte-faithful plistlib reconstructions of the real archive).

`SpeechPreprocessor` (Sources/Audio/SpeechPreprocessor.swift) runs inside `SpeechEngine.speak()` on every read: **sanitize** (strips control chars, invisible Unicode — ZWSP/BOM/variation selectors/directional marks — private-use chars; emoji ZWJ sequences survive; NBSP → space) → **hash abbreviation** (word-bounded all-hex runs at standard digest lengths — 32/40/64/128 → md5/sha1/sha256/sha512, named by length convention — collapse to "md5 ending in 2 7 e"; requires ≥1 digit and ≥1 letter so numeric IDs don't mislabel; runs at every level, `verbalizer.hashes: false` disables) → **identifier splitting** (`splitIdentifiers` — camelCase humps and internal snake_case underscores become spaces: "parseHTMLBody_v2" → "parse HTML Body v 2", acronym runs split before a following word — XMLHttp → "XML Http"; a token qualifies only with an internal underscore or a case hump alongside lowercase, so plain words, ALLCAPS acronyms, UTF16-style tokens, and `__init__` dunders pass untouched; own toggle like hashes — every level, `verbalizer.identifiers: false` / `:config identifiers off` disables) → **verbalize** (speaks symbols as words per `verbalizer.level`: `none`/`some`/`most`/`all`, default `most` = code symbols spoken, prose punctuation left to natural pausing; digraphs like `->` "arrow" and `!=` "not equals"; symbol runs ≥3 collapse to "N name" using the full symbol table even for symbols unspoken at the level — `-----` says "5 dash" at most; a bare `...` stays raw below `all` (natural ellipsis pause); unnamed symbols cap at 3 raw) → **whitespace normalize**. Sanitize is the anti-bail stage and runs at every level. Empty-after-processing skips speaking but still fires the `completion:` (the inline CLI blocks on it). `announce()` gets sanitize only. Config: `verbalizer` block — `level`, plus `symbols` overrides (`{"*": "asterisk", "->": "maps to", "%": ""}`; empty string silences a symbol). Config is read at daemon init; `:config level|hashes|identifiers` rebuilds the preprocessor live, while hand-edits to the file still require a restart. Tests: `swift test` (Tests/MardukTests, macOS only).

## Audio Ducking Flow

`prepareToDuck()` (before speech) → probes the CoreAudio-based mediaKey state synchronously (so our own speech can't contaminate it), stores it on the ducker queue → `duck()` (async, on didStart) → uses cached state, probes Apple Music/Spotify via AppleScript on the ducker queue (never on the main thread), only pauses what's actually playing → `unduck()` (async, on didFinish/didCancel) → only restores what was ducked. `audioProducingPIDs()` filters out our own PID AND Apple's speech-synthesis service process (enhanced voices render there — without the filter, back-to-back reads mistake draining speech audio for playing media). **The media key is only sent to a claimed room:** `.mediaKey` counts as playing ONLY when an audio-producing process is a media-key CLIENT (`audibleMediaKeyClientExists` — bundle-ID list ∪ executable-path tokens, because browser audio renders in helpers: Chrome → "Google Chrome Helper", Safari → com.apple.WebKit.WebContent; extras via `ducking.mediaKeyApps`). An UNCLAIMED play/pause press makes macOS launch Music (first-external-user report) — calls/games/unknown audio are therefore spoken over, never paused. All osascript runs have a kill-on-timeout watchdog so a hung System Events can't wedge the ducker queue.

**Important:** Never use fixed delays to wait for speech to finish. Pass a `completion:` to `speech.speak`/`speech.announce` to chain actions after speech completes — this ensures unduck() has time to restore media before the next action. Completions are tracked per-utterance (fired on finish or cancel, only for their own utterance), so a replaced utterance can never fire or clear its successor's callback. Only the current utterance's end unducks; stale didCancel deliveries are ignored.

## Keyboard Modal System

- **NORMAL** (default): Commands active, letter keys suppressed. `s` toggles MARDUK-NATIVE pointer hover speech (`HoverSpeech`, Sources/Input/HoverSpeech.swift): dwell ~0.35s → AX element-at-position → speak the element's NAME (title ?? description ?? value, minimal verbosity by founding rule) in the READING voice/rate/pitch over the echo channel — no Settings shortcut setup, no media pausing, never disturbs a read; same-element/same-label repeats and Marduk's own windows stay silent; deactivated on Ctrl+Option+M off and teardown. (Replaced the old macOS speak-items-under-pointer shortcut integration, whose separately-configured system voice never matched.)
- **Typing rescue** (NORMAL): unmodified letter keyDowns are withheld ~300ms (`keyboard.typingBurstMs`; `keyboard.typingRescue: false` disables) instead of executing immediately. A quick burst containing a non-command letter means the user forgot the mode and is typing — auto-switch to INSERT, play a falling sweep (`Earcon.fallToInsert`), and replay the withheld keys synthetically as down+up pairs (marker-tagged, in order — pairs because the real keyUps usually passed through before the replay posts; a replay-rollover guard swallows and re-queues keys typed during the switch so nothing reorders). All-command bursts (`s` then `r`) execute as commands on timer expiry via recursive redispatch through `handleEvent` (mode-aware: a buffered `i` flips to INSERT and later buffered keys get typed, not executed). Consequences: single letter commands fire ~300ms late; `i` on an empty buffer is instant (the i-then-type flow); `v`+motion (`vj`, `v3j`) flushes immediately and stays instant; `tt` resolves on the second t (commands buffered before the pair flush first); words starting with `i` lose their leading i; a command followed quickly by `k` reads as typing (words like "skip" must rescue); with rescue disabled there is no `tt` (each `t` speaks time immediately).
- **INSERT** (`i`): Entry always plays `Earcon.fallToInsert` (explicit `i` and typing rescue sound identical). All keys pass through. Escape is tap/hold: a tapped Escape is delivered to the app on key release (synthetic re-post — vim/Claude Code keep their Escape); holding it past the threshold (default 400ms, `keyboard.escapeHoldMs` in config) returns to NORMAL and plays a rising sweep (`Earcon.riseToNormal`) — the app never sees a held Escape. The withheld press is flushed early if another key rolls over it, so fast Esc+j typing stays ordered. Modified Escapes (Cmd/Ctrl/Shift) pass through untouched.
- **VISUAL** (`v`): Character-wise selection with hjkl, `r` reads selection. Entry is spoken ("visual"/"visual line"); exit via Escape/`v` plays `Earcon.riseToNormal` (`r` exits silently — the read is the feedback)
- **VISUAL LINE** (`V`): Line-wise selection, auto-selects current line on entry
- **COMMAND** (`:` = Shift+; in NORMAL): vim-style command line, driven entirely by the event tap. Chars append to `commandBuffer` (keycode→char via `commandKeyChars`; echo per char when `commandEcho`), Return submits via `onCommandSubmit` (empty = cancel), Escape cancels, Delete edits (exits on empty buffer), Tab autocompletes to the palette selection via `onCommandTab`→`replaceCommandBuffer`, Up/Down move the selection (Ctrl+N/Ctrl+P are vim-native synonyms — the second Ctrl carve-out, COMMAND-only; passing them through would emacs-move the app's cursor mid-command), `?` speaks the current options, a ~1.5s typing pause speaks them automatically (`onCommandIdle`; silent when none), and `/` on an empty buffer starts an fzf-style FUZZY SEARCH over the whole catalog (commands + config keys; greedy-subsequence `fuzzyScore`, lower = tighter; Return accepts the selection — commands execute, keys expand into the value stage; `/` buffers never auto-resolve). Cmd/Ctrl AND Option combos pass through (the user's zoom shortcuts ride on Option — command mode must never eat them), as do unmapped non-typing keys (F-keys, keypad, custom codes); only unmapped typing-shaped keys (`typingPunctuationKeys`) buzz, logging the keycode. Autorepeat suppressed except Delete/arrows. Commands (unique-prefix expanded, vim-style — `:conf ra 230` works): `:help`/`:h`, `:commands`/`:c`, `:tutorial` (toggles the interactive tour — Tutorial.swift watches real mode/read/pause events), `:tip` (random feature tip from HelpText.tips, no immediate repeats), `:config`/`:set <key> <value>` (live apply + persist), `:pronunciation` (opens the system pronunciation editor pane — see System pronunciations), `:voices` (voice picker: expands to a `voices ` stage listing installed English voices best-quality-first; typed text fuzzy-filters; arrow selection previews each candidate IN that voice via `announce(_:voice:)`; Return/`/`-accept applies to `speech.voice` + persists `voiceIdentifier`; the tap's Return handler keeps COMMAND mode alive for `voices`-prefixed buffers exactly like `/`, and the daemon intercepts them before parse — `.voices` in the parse switch is compiler-exhaustiveness only). **Auto-accept (dmenu semantics):** ~350ms after the last typed char (`ColonCommand.autoResolve`, debounced in `handleCommandChange` so fast full-word typing is never cut off), an unambiguous buffer acts without Enter — argless commands and unique enum values execute, `config`/keys expand and advance; numbers always need Enter; deletions never auto-accept (canAutoAccept flag on `onCommandChange`), or removing an auto-added space would re-add it. Settings table lives in `ColonCommand.settings` (shared by parser, palette, and validation). The **command palette** (Sources/Display/CommandPalette.swift, gated by `keyboard.commandPalette`) is a dmenu-style panel whose INPUT still comes from the event tap — it renders the buffer + candidates from `CommandCompleter` — but it TAKES key focus while open (Spotlight-style: remembers `NSWorkspace.frontmostApplication`, `NSApp.activate()`, `makeKeyAndOrderFront`; borderless panels need the `canBecomeKey` override) and reactivates the previous app on hide. Position is configurable (`keyboard.palettePosition` / `:config position pointer|center`): `pointer` (DEFAULT) opens at the cursor — the ONLY zoom-proof placement, since zoom always keeps the real cursor in view; `center` is fully centered on the pointer's screen. Zoom focus-following can NOT be made to track the palette — exhaustively disproven (bare binary, signed bundle, .regular policy, real focused caret, every AX notification, every form of synthetic pointer input; no viewport API exists — the old border overlay's lesson). Do not reopen that investigation. The prompt line is a REAL focused `NSTextView` (PromptTextView — keyDown/insertText swallowed, the tap owns input) with a genuine blinking caret, so zoom's follow-focus/text-insertion modes track the palette natively via AX notifications; level is `.floating`, NOT `.screenSaver` (the aggressive level broke zoom state while open); rows are mouse-clickable (click = Tab on that row, `onRowClick`). With the palette off, command mode is pure audio and steals nothing. `DaemonServer.run()` initializes `NSApplication.shared` (`.accessory`) for it.
- **Always-active:** Ctrl+Option+M toggles Marduk (feedback per `keyboard.toggleSound` / `:config togglesound speech|earcon` — spoken "Systems engaged/disengaged" by default, bloop earcons opt-in), Option+Escape speaks selection/stops speech — NOTE: macOS's own Speak Selection defaults to the SAME shortcut and its hotkey handler fires upstream of the tap (can't be suppressed programmatically). The fix is the KARABINER PROFILE HANDOFF (Marduk strongly assumes Karabiner): the daemon owns the profile NAMED "Marduk" — adopted as-is if the user made one (only Marduk's own rule, matched by description, is refreshed inside it), bootstrapped as a clone of the selected profile otherwise (`activateKarabinerProfile`: backup to karabiner.json.marduk-backup before every rewrite, parse failure = no write, non-Marduk profiles never touched; selection via karabiner_cli, NOT the flaky file-watch reload). The injected rule routes the read button (`keyboard.karabinerReadKey`, default equal_sign = Naga 12) — DEVICE-SCOPED via `keyboard.karabinerReadVendorId` (default 5426 = Razer; 0 = any device) + optional `karabinerReadProductId`, on BOTH manipulators, because equal_sign is also the real keyboard's =/+ key and an unscoped rule ate it in every mode (field report) — to Ctrl+Option+Escape — the read handler accepts extra modifiers on Option+Escape, deliberate — while `marduk_up` is 1, plain Option+Escape (macOS Speak Selection) otherwise. EVERY exit hands back the user's profile: clean stop/disable directly, SIGTERM via a sig_atomic_t flag the run loop polls into full teardown, hard crashes via `mardukCrashRestore` (async-signal-safe: pre-built argv + posix_spawn + raise, armed by `armCrashRestore`); only SIGKILL/power loss escapes and KeepAlive's relaunch heals it. `setKarabinerVariable` publishes marduk_up on the same lifecycle (standalone rule for manual setups: assets/karabiner/marduk-read-button.json) — three-way since Escape-tap-pause exists: audibly SPEAKING → stop only; PAUSED (silent, still reports isSpeaking — `isReadPaused` closure) → stop the old read AND read the selection in the same press (a paused read must never silently eat the press); idle → read selection — all three selection reads (Option+Escape, `r` post-triple-click, visual `r`) go through `readSelection`: AX `kAXSelectedTextAttribute` first, synthetic-Cmd+C clipboard fallback when AX errors or returns empty (huge Cmd+A selections blow the 0.5s AX timeout; Firefox text boxes/iMessage never expose it). AX failures log their error code; -1001 = the timeout
- **Firefox narration handoff (`n`, `8`):** in NORMAL with Firefox frontmost, `n` stops Marduk's speech, pauses media via the ducker, sets a **duck hold** (`AudioDucker.holdDucking` — unduck() becomes a no-op so Marduk's own announcements can't resume music mid-narration), then posts a synthetic `n` to Firefox (~0.35s later) which drives Reader-mode Narrate's play/pause (`startNarrationHandoff`). Second `n`, Escape (NORMAL), or Ctrl+Option+M posts `n` again and `releaseHoldAndUnduck()`. STARTING requires `narrationContext()` — an AX walk UP from the focused element to the nearest AXWebArea checking for an `about:reader` URL (main queue only, 0.5s timeouts); this is what makes INSERT-mode `n` safe: it always passes through (it IS Narrate's toggle) and only duck/holds when focus is inside a Reader document, so text boxes and the URL bar keep their n's. `8` (NORMAL, Firefox, keycode 28 — the one exception to digits-pass-through) is the combo key: posts Cmd+Option+R, polls `narrationContext()` (~2.5s), then hands off; `8` while narrating = stop + release + close Reader. Frontmost app is CACHED via an NSWorkspace didActivate observer (`frontmostBundleID`) — never queried in the tap callback. `n` is a command letter ONLY when Firefox is frontmost (`isCommandLetter()`), so typing rescue still treats "sun"/"runs" as typing elsewhere; hold order matters — set the hold BEFORE speech.stop() so the cancel's unduck lands blocked. Known limit: state can desync if the user drives Narrate from Firefox's UI mid-handoff (n is a blind toggle).
- **Speed keys (opt-in, OFF by default):** `:config speedkeys on` / `keyboard.speedKeys` — Option+Up/Down nudge the speech rate ±10 WPM in NORMAL/VISUAL only (INSERT/COMMAND excluded: apps own Option+arrows there; Shift combos pass through). Deliberately NO autorepeat guard — holding glides. Rate applies per event; announcement + config write debounce 500ms after the last nudge (`rateSaveTimer`). Setting key is `speedkeys`, NOT `ratekeys` — `rate` would be its prefix, breaking unique-prefix expansion (guarded by `testNoSettingKeyIsPrefixOfAnother`).
- **Visual mode overlay (opt-in, OFF by default):** `ModeOverlay` (Sources/Display/ModeOverlay.swift) — screen-edge border and/or pointer-following dot colored by mode: red NORMAL, green INSERT, blue VISUAL, purple READING (`overlay.readingColor`, default #AF52DE — READING capture OVERRIDES the mode color via `setReading`, driven by `KeyboardMonitor.onReadingChange` from `readingCapture`'s didSet, and `rebuildOverlay()` re-applies it so a live `:config thickness` mid-read stays purple); COMMAND keeps normal's color (the palette is its own indicator); `"none"` as a color hides that mode. `:config border on|off`, `:config pointer on|off`, `:config thickness <1-40>` apply live (`rebuildOverlay()` — full teardown + recreate); colors and `pointerSize` are file-only (`overlay` config block, all keys Optional). Marduk-off hides everything (`setEnabled` via `onEnabledChange`). Zoom caveat: fullscreen zoom magnifies the composited framebuffer, so the border can be entirely out of the zoomed view — the pointer dot is the zoom-proof indicator (zoom always tracks the pointer). Windows are `.screenSaver` level, click-through, never take key focus (safe, unlike the palette).
- **Space (read pause/resume):** while a content read is speaking or paused (`SpeechEngine.readActive` — announcements excluded), unmodified Space in NORMAL/VISUAL pauses IMMEDIATELY / resumes (never pauseSpeaking(at:.word) — a deferred pause landing after a stop wedges the synthesizer mute and kills didStart-driven ducking; field-diagnosed), and never reaches the app. In INSERT, Space is always a real space, even mid-read. Otherwise Space types/passes as normal. A typing-rescue burst in flight wins over pause (user is typing). Escape in NORMAL cancels a paused read (a paused synthesizer still reports `isSpeaking`), freeing Space back to normal. Media stays ducked/paused across a speech pause — only the read's end or Escape (stop) unducks. `readActive` is plain stored state so the tap callback can read it synchronously (no AV query in the callback).
- **Read motions / READING capture (ON by default; `:config readmotions off` / `keyboard.readMotions` disables):** while enabled, an active read (`readActive`) CAPTURES the keyboard from ANY mode (`readingCapture`, set synchronously by `SpeechEngine.readActive`'s didSet → `onReadActiveChange` → `readStateChanged`; entry skipped in COMMAND — post-`:` confirmation reads must not steal the palette). READING is a real mode: `i` and held Escape are the ONLY exits — `i` stops the read → INSERT with the falling sweep; Escape is tap/hold (`pendingReadingEscape`, same `escapeHoldThreshold` + `escapeHoldFired` keyUp-swallow as INSERT's): a TAP toggles pause/resume exactly like Space (resolved on keyUp; a rollover key resolves it as a tap first), a HOLD stops the read → NORMAL with the rising sweep; every other typing-shaped key (letters incl. `k`, stray digits, punctuation, Return/Tab/Delete) buzzes instead of leaking into the app — a read fired from INSERT used to let `(`/`gg` type over the selection. Cmd/Ctrl/Option combos and non-typing keys (arrows, F-keys, Naga codes) pass. A natural read end just drops the capture — the underlying `mode` is never touched, so the user lands back exactly where they were (INSERT stays INSERT); a typing burst in flight when the read starts wins (`declareTyping` drops the capture); `readMotionsEnabled=false` and Ctrl+Option+M also drop it. Keys: `r` abandons the read and reads the paragraph under the pointer instead (`readAtPointer`, shared with the NORMAL `r`; the replacement keeps media ducked and the capture engaged), `R` — CONTINUOUS READING, in NORMAL and in capture — reads the focused document to the end (`readDocumentFromCaret`: AX `kAXValueAttribute` full text; START = the char under the mouse POINTER when it's over the element's text — parameterized `AXRangeForPosition` (`textOffsetAtPointer`, top-left-origin coords flipped from NSEvent's), else `pointerRowEstimate` (pointer's vertical fraction of the element frame picks a line inside `kAXVisibleCharacterRange` — Terminal doesn't answer range-for-position), else the caret via `kAXSelectedTextRangeAttribute`, else 0; snapped to `wordStart`. ## Display Management (DisplayInverter)

`DisplayInverter` (Sources/Input/DisplayInverter.swift, `[display]` logs) — dark-mode workflow support; the user lives in dark mode, so inversion is a SCOPED SACRIFICE: while a bright app is front it goes dark (and the rest of the world goes light, unwatched); leaving reverts. Final architecture after a full field campaign — every earlier scheme is listed at the end as a PROHIBITION, do not rediscover them. (1) **App-list inversion** (`:config invert on|off` master, default on): BUILT-IN prefixes (`builtInInvertPrefixes`: com.netacad.PacketTracer any-version, com.apple.iWork.Pages — user-ratified hardcoding) + `display.invertForApps` exact IDs (file-edited). **Mechanism (the ONLY one that sticks, hardware-settled):** fire the system Invert Colors keyboard shortcut VIA SYSTEM EVENTS osascript (`applyInversion`, kill-on-timeout watchdog; needs the Automation grant Marduk already holds). The user's real binding auto-discovers from `com.apple.symbolichotkeys` entry 21 (params[1]=keycode, params[2] mask bits 17/18/19/20 = shift/ctrl/opt/cmd; fallback Shift+Cmd+F13; disabled shortcut → log hint). The hotkey is a TOGGLE — only ensureInverted's transition guard plus the TOGGLE LOCKOUT (1.5s) make it safe (one change, then silence; deferred wants converge via the heartbeat). **Speed (user-tuned after the mechanism proved to stick):** listed apps and Preview dark-PDF act on the RAW activation event (no dwell — eager inversion is always safe); dwell 0.4s gates only auto-measurement; heartbeat 1s; revertAfter 3s (floor = PT's post-inversion rebuild flap, ~1-2s of churn caused by the display change itself — physics, not paranoia); settle 0.6s. **State machine:** events (0.8s dwell, dual-signal `resolveFront` biased toward listed apps, disagreements logged) may INVERT — never revert. Reverting is the HEARTBEAT poll's monopoly: the holder (listed or auto-bright app that earned the inversion) must merely be seen front once per revertAfter to keep it; sitting still renews every beat. Bookkeeping seeds from `com.apple.universalaccess/whiteOnBlack`; verification NEVER acts (resync + hint only). (2) **Preview dark PDFs** (`:config pdfdark auto|on|off`, default AUTO = follow system appearance, live theme-flip reaction; going light never un-darks open windows): presses Preview's dark-appearance menu item via the AX menu bar, found by SEARCH ("dark appearance" then "dark mode" — macOS 26 titles it "Use Dark Appearance for PDF"), checkmark consulted first; per-window observer for windows opened while front. English menus only. (3) **Brightness auto-detection** (`:config autoinvert`, default off; Screen Recording TCC — often granted SILENTLY, flow opens the pane): 64px SCScreenshotManager capture of the LARGEST window, `meanBrightness` vs `display.autoInvertThreshold` (percent, default 70), hysteresis −0.08 while inverted; bright verdicts invert (holder=bundle), dark verdicts DO NOTHING. Judges only UNLISTED apps; Preview skipped when pdfdark covers it; capture failure logs once. **PROHIBITIONS (each failed in the field, Qt AND native apps):** raw CGEvent chords for the hotkey (symbolic-hotkey handler ignores every shape, real modifiers included); private `UAWhiteOnBlackSetEnabled` (SELF-FLICKERS on macOS 26 — filter flashes on, universalaccessd overrides it off, pref stranded — proven by the one-shot-latch experiment); event-driven reverts in any costume (immediate, dwell-only, settle-window, window-stack trust, two-strike, lease); verification retries and coherence pulses (acting twice inside one transition IS the flicker); trusting instants over envelopes. The window signal (CGWindowList layer-0) can miss PT entirely; NSWorkspace activation flaps under PT churn — consult both, bias listed, decide by envelope. Tests: meanBrightness (VisualFollowTests neighbors).

## Release Pipeline

`scripts/release.sh <version>` (run on the Mac, repo root) produces a notarized downloadable release: bumps Version.swift → commit/tag/push → **waits for CI green** (`gh run watch`) → `swift build -c release` → `marduk bundle` (internal CLI command: assembles Marduk.app from the invoking binary, prints the exec path) → **distribution re-sign** (the Bundler's own signature is development-grade `--timestamp=none`; notarization requires `--options runtime --timestamp` + the `com.apple.security.automation.apple-events` entitlement so media control survives the hardened runtime) → notarize the APP (zip submit, `marduk-notary` profile) → staple the app → build a drag-to-Applications DMG (`hdiutil`, /Applications symlink) → sign + notarize + staple the DMG too (belt-and-braces: both artifacts verify offline) → `gh release create` with the DMG and commit-subject notes → **update the Homebrew tap** (`spencer-dollahite/homebrew-marduk`): rewrite `Casks/marduk.rb` with the new version + local DMG sha256 and push (runs before the local DMG is deleted; the cask pins the VERSIONED asset URL `releases/download/v<version>/Marduk.dmg`, never the floating `latest` link — hash pinning breaks otherwise). Users: `brew install --cask spencer-dollahite/marduk/marduk`. **Release installs** (bundle outside any repo): `marduk install` detects `/Marduk.app/Contents/MacOS/` with no project dir and installs AS-IS (reassembling/re-signing would destroy the notarized signature); the update paths detect no-project-dir and check the latest GitHub release tag instead (`checkLatestRelease` + pure `ReleaseCheck.parseLatestTag`, unit-tested): manual `u` speaks "Marduk X is available" / "up to date", the periodic timer announces each new version once (`lastAnnouncedRelease`); the install hint is channel-aware — `InstallChannel` detects Homebrew via the Caskroom dir and says "run brew upgrade", plain releases get "download from GitHub" (release-channel self-INSTALL = future work). `:bug` prefills the issue form's setup field with version + channel + macOS version. **Finder onboarding:** double-clicking Marduk.app (no CLI args, bundle path, no repo) runs `install`; if already running it speaks "already installed"; after install with no Accessibility grant it opens the Privacy_Accessibility Settings pane while the daemon's spoken self-heal explains the grant. Download→drag→double-click→follow-the-voice is the entire release onboarding.

## Running as a Service (launchd)

`Bundler` (Sources/App/Bundler.swift, `[bundle]` tag) assembles `<repo>/Marduk.app` on the Mac at install/update time — Info.plist via PropertyListSerialization, the M icon DRAWN IN CODE (CoreGraphics, same path as assets/logo.svg, iconutil-packed; failure = iconless bundle, non-fatal), binary copied in, bundle signed, then ATOMIC SWAP (`Marduk.app.new` → aside → in; never modify the live bundle in place — a running daemon keeps its inode). **Identity trio, keep identical:** launchd label == `Codesign.identifier` == CFBundleIdentifier == `com.marduk.daemon` (identical designated requirement is what can carry TCC grants across the bare-binary→bundle migration). `LSUIElement` keeps it out of the Dock. Bare-binary operation (`start --foreground` debugging, binary copied out of the repo) still works everywhere — bundle-ness is detected, never assumed. SMAppService adoption = future work.

**Migration (bare binary → bundle):** detected by `LaunchAgent.installedProgramPath() != bundleExec`. The CLI update rebootstraps synchronously (safe — it isn't the daemon). The daemon-side `uu` CANNOT bootout itself (deadlock: launchd waits for our exit while we wait for launchctl) — it writes the new plist FILE only, spawns `LaunchAgent.relaunchDetached()` (a `/bin/sh` helper via `posix_spawn` + `POSIX_SPAWN_SETSID` — mandatory, because the OLD plist lacks `AbandonProcessGroup` and launchd SIGKILLs the daemon's process group on exit), announces the one-time migration (even on the silent auto-update path), and exits CLEAN (0) so KeepAlive can't race-relaunch the old path before the helper's bootout→bootstrap→kickstart lands. New plists carry `AbandonProcessGroup: true`.

`LaunchAgent` (Sources/App/LaunchAgent.swift) manages a per-user LaunchAgent: label `com.marduk.daemon`, plist at `~/Library/LaunchAgents/com.marduk.daemon.plist`, stderr/stdout to `~/Library/Logs/marduk.log` (truncated on install/start if >10 MB, and unconditionally on every SUCCESSFUL update just before restart — fresh log per build, which also ages out anything older logging policy wrote; update FAILURES never truncate, preserving their diagnostics; no rotation). The plist runs `Marduk.app/Contents/MacOS/marduk start --foreground` so launchd supervises the daemon process directly. `KeepAlive = {SuccessfulExit: false}`: crash or non-zero exit → relaunch (launchd throttles rapid loops ~10s); clean exit 0 (`marduk stop`) stays stopped until next login or `marduk start`. When the agent is installed: `start` kickstarts the service (re-bootstrapping if needed), `update` (CLI) kickstarts instead of spawning, and the daemon-side `u` hot-update exits with code 75 after clean teardown so launchd relaunches the freshly built binary — a `Process` spawn would be unsupervised. `marduk start` exits 0 when the daemon is already running: under KeepAlive a non-zero exit from the agent-spawned instance would relaunch-loop. **TCC / signing:** install and both update paths sign the assembled BUNDLE (`Codesign.sign(bundleAt:)` — one call on the staging .app seals the nested executable; only ever called on `Marduk.app.new`, so no in-place hazard; `sign(binaryAt:)` remains for the bare-binary fallback and keeps its copy-sign-swap). Identity is certificate + `com.marduk.daemon`, now also the CFBundleIdentifier — the grant is bundle-anchored and survives rebuilds. If no keychain identity exists, signing is skipped with a loud log (plain `swift build` also skips assembly/signing — deploy via `marduk update`). If the tap dies after an update anyway, the daemon announces it and re-adding the Accessibility entry (pick `Marduk.app` in the file dialog) heals it live.

## Runtime Files

- `$(getconf DARWIN_USER_TEMP_DIR)marduk.sock` — daemon IPC socket (per-user /var/folders/…/T/; `MardukDaemon.runtimeDir` via confstr, NOT /tmp — world-writable /tmp let any local user drive the socket or squat the paths and block startup). Socket is chmod 0600 and `handleClient` getpeereid-rejects other UIDs. CLI and daemon share the path constants, so an updated CLI can't reach a pre-update daemon until its restart — one-time blip, `marduk start` heals.
- `$(getconf DARWIN_USER_TEMP_DIR)marduk.pid` — daemon PID file
- `~/.config/marduk/config.json` — user configuration (`.welcomed` / `.ke-noticed` markers live beside it — first-run welcome and the once-only no-Karabiner onboarding hint; `marduk status` also reports Karabiner presence)
- `~/Library/LaunchAgents/com.marduk.daemon.plist` — launch agent (when installed)
- `~/Library/Logs/marduk.log` — daemon log (when running under launchd)
- `<repo>/Marduk.app` — assembled bundle (gitignored; rebuilt by install/update)
